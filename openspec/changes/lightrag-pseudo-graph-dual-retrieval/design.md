## Context

MemoHub 现状是“双轨（GBrain/ClawMem）+ LanceDB 向量检索 + MCP 工具封装”。在实际链路里：

- 主项目（`src/lib`）已经在记录结构里引入了 `entities` 与 `hash`，并在代码轨写入时做了实体抽取；知识轨写入时默认空数组。
- MCP Server 侧存在“首次建表 seed 与字段不一致”的风险：若用户只启动 MCP 来初始化库，表结构可能缺失 `entities/hash/access_count/last_accessed` 等列，且数组字段类型推断可能不稳定，从而影响实体过滤与后续写入兼容性。
- 检索能力以向量为主：跨轨检索由 MCP 的 `search_all/memo_search` 提供，但内部仍是“各轨向量召回 + 合并”，缺少宏观上下文扩展与统一的流水线编排。
- “全文检索/FTS”在类型枚举里出现过，但仓库里未落地实际执行通道；因此本次设计把 Lexical/FTS 视为可插拔通道，优先落地轻量词法召回与实体扩展。

本次变更目标是：在不引入重型图数据库与复杂任务引擎的前提下，落地 LightRAG 式“伪图谱 + 双层检索”，并用可组合的三段式检索流水线把跨轨联动工程化。

## Goals / Non-Goals

**Goals:**

- 伪图谱：以 `entities: string[]` 作为统一锚点，支持跨轨的宏观关联召回（不引入图数据库）。
- 双层检索：Low-level（向量 TopK 保细节）+ High-level（高频实体扩展召回补全上下文），并可配置上限以避免查询膨胀。
- 三段式检索流水线：Pre（意图/实体）→ Exec（多路召回）→ Post（融合去重/排序/引用输出），保证行为一致、可调参、可测试。
- 检索阶段跨轨联动：不做写入阶段显式 cross-link；通过实体扩展与统一融合排序实现跨轨互证。
- 强制引用输出：MCP 返回结果必须携带可追溯字段（至少 `trackId`，以及代码侧 `filePath/symbol_name`），为上层 Agent 提供 citation-ready 数据。
- 兼容与可迁移：对既有数据尽量零破坏；新增能力可通过降级策略在旧库/旧 schema 下尽可能工作。

**Non-Goals:**

- 不在本次变更引入“文件系统 CAS（灵肉分离）”与复杂治理管道（冲突检测、澄清闸门、Wiki 生成等）；这些可作为后续 P2/P3。
- 不引入 LangChain/NetworkX/专用图数据库/复杂 DAG 任务流引擎。
- 不强依赖 LanceDB 原生 FTS 功能一次到位；FTS 作为后续可替换/增强通道。

## Decisions

1) **统一记录契约：entities/hash/citation 字段在两轨一致**

- 选择：将 `entities: string[]` 视为“伪图谱锚点”，`hash` 作为幂等去重与融合去重主键之一；引用类字段（如 `trackId/filePath/symbol_name/timestamp`）作为检索输出的强约束。
- 理由：entities 是“宏观关联”的最小结构单元，hash 是“重复污染”的主要防线；引用字段是“减少幻觉”的必要条件。
- 影响：需要修复 MCP 侧首次建表 seed，保证 schema 与 `src/lib` 对齐；MCP 的写入工具需要补齐 `entities/hash` 写入（至少不缺列）。

2) **双层检索实现为：向量召回 → 高频实体抽取 → 实体扩展召回**

- 选择：Exec 阶段先做各轨向量 TopK，然后在 TopK 结果集合上做 entities 频次统计，选取 TopE（可配置，默认 3-8）进行扩展。
- 理由：用 TopK 结果来“自举”实体集合，避免对 query 做过重的实体抽取依赖；同时把扩展范围绑定在当前 query 的语义邻域内，可控且容易解释。
- 可解释性：扩展召回必须在返回里携带“触发实体列表”与“命中实体数”等可观测字段（内部使用，必要时可透出给上层）。

3) **实体扩展召回的执行策略：DB 过滤优先 + entity_index 兜底**

- 选择：优先尝试在 LanceDB `.where(...)` 使用数组包含过滤（例如 `array_contains(entities, 'X')` 的等价表达式），若不可用或不稳定，则启用派生索引 `entity_index`。
- 理由：DB 过滤实现最简、性能最好；但表达式支持存在不确定性，必须提供兜底以保证上线可控。
- entity_index 形态：以“派生索引而非显式 cross-link”为原则，仅由 `entities` 自动生成，用于 entity → record_id 的扩展召回；不承担“关系语义”，不引入一致性地狱。

4) **检索流水线以“可组合阶段函数”实现，而不是先上事件总线/任务引擎**

- 选择：定义 `PreResult / ExecResult / PostResult` 的 TypeScript 契约，每段实现为纯函数（或近似纯函数），在 MCP 内部组装成 pipeline。
- 理由：更符合当前代码结构（CLI/MCP 直接调用类方法），可测试性强、可逐步替换；后续要做 IoC/Provider 化时，只是把阶段函数装配到不同的 engine 里。

5) **多路召回：Vector 为主，Lexical 先轻量落地**

- 选择：本次将 Lexical/FTS 通道定义为可插拔执行通道；第一期用“轻量词法召回/加权”（例如 token overlap / 关键字命中）补齐向量对符号/路径的敏感度；后续再替换为真正的 FTS 实现。
- 理由：避免被“FTS 选型与底层实现差异”阻塞整个检索升级。

6) **融合排序策略：向量分 + 实体覆盖分 + 轨道权重 + 新鲜度**

- 选择：Post 阶段对候选集合按综合分排序：
  - 向量相似度（基础）
  - entity 覆盖度（命中扩展实体的数量/权重）
  - track 权重（按意图：偏代码时 ClawMem 更高，偏知识时 GBrain 更高）
  - 时间衰减/新鲜度（可选，默认弱影响）
- 理由：兼顾“细节命中”和“宏观上下文”，并为不同检索意图提供稳定的偏置。

## Risks / Trade-offs

- **[Risk] MCP/CLI schema 初始化不一致导致 entities 列缺失或类型错误** → 统一 seed 记录字段与非空数组推断策略，并提供“schema 校验/修复（必要时重建或迁移）”的最小脚本/指引。
- **[Risk] DB 侧数组包含过滤不可用** → 采用 entity_index 兜底，不依赖底层表达式能力；同时通过参数限制扩展召回规模。
- **[Risk] 查询膨胀（扩展召回拉太多）** → 引入硬上限（TopK、TopE、最大扩展候选数、跨轨合并最大条数）并在 Post 阶段强去重与截断。
- **[Risk] 多套检索入口结果不一致（search_all vs memo_search 等）** → 明确“统一流水线”的单一实现入口，其他工具逐步改为调用同一 pipeline 或标记为兼容层。
- **[Trade-off] 不做写入阶段显式 cross-link 会损失部分“精准关联”** → 先用检索联动验证价值；后续若确有需求，再在不破坏实体索引的基础上增量引入显式链接（作为可选能力）。
