# Multi-Session State Sharing

## Why

Phase 0 和 Phase 1 实现了单会话的事件摄取和查询。为了让多个 Hermes 和 IDE 会话能够共享项目和任务上下文，我们需要实现多会话状态管理。

**这是一个复杂的分布式系统问题，需要仔细设计并发控制和冲突解决机制。**

## What Changes

### 新增功能

- **SessionState 读模型**: 在 Soul 中添加 `session_states` 表，存储活跃会话/任务/项目状态
- **状态聚合**: 跨会话聚合项目和任务状态
- **冲突检测**: 检测同一会话/任务的状态冲突
- **状态查询**: 通过 MCP 工具查询多会话状态

### 分阶段实现

#### Phase 2.1: 只读多会话感知
- 可以查询其他会话的状态
- 不做状态合并或冲突解决
- 每个会话状态独立存储

#### Phase 2.2: 写时复制 + 后台合并
- 每个会话有自己的状态副本
- 后台 Librarian 负责合并
- 用户可以手动解决冲突

#### Phase 2.3: 实时状态同步
- WebSocket 或轮询机制
- 最终一致性保证
- 自动冲突解决

## Capabilities

### 新增 Capabilities

- **session-state-manager**: 管理会话状态生命周期
- **state-aggregator**: 跨会话聚合状态
- **conflict-detector**: 检测状态冲突
- **conflict-resolver**: 解决状态冲突

## Non-Goals

- ❌ 实时协作编辑
- ❌ 完整的分布式事务
- ❌ 强一致性（最终一致性即可）

## Success Criteria

- [ ] 多个会话可以读取共享的项目状态
- [ ] 状态冲突能被检测和解决
- [ ] 性能满足预算（< 200ms P99）
- [ ] 单元测试覆盖率 > 80%

## Risks

| 风险 | 缓解措施 |
|------|----------|
| 分布式状态复杂度高 | 分阶段实现，从简单开始 |
| 并发冲突频繁 | 使用最终一致性，人工解决 |
| 性能开销大 | 缓存热点状态，异步聚合 |
